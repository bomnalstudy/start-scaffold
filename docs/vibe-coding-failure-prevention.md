# Vibe Coding Failure Prevention Plan

이 문서는 바이브 코딩이 "충분히 괜찮은 상태"에 도달했는데도 AI가 멈추지 못하고 프로젝트를 이상한 방향으로 끌고 가는 문제를 막기 위한 기획안입니다.

핵심 목표는 하나입니다.

- AI가 문제를 끝내는 능력보다 문제를 계속 만들 가능성이 커지는 시점을 빨리 감지하고 멈추게 한다.

## 배경

바이브 코딩 AI는 기본적으로 "문제를 해결하려는 방향"으로 계속 움직입니다.

문제는 이 성질이 아래 상황에서 오히려 독이 된다는 점입니다.

- 이미 본래 목표는 거의 달성했는데 사소한 불편을 계속 최적화함
- 현재 눈앞의 에러만 좇다가 원래 해결하려던 문제를 잊어버림
- 특정 테스트나 특정 케이스만 통과하는 뾰족한 해법으로 빠짐
- 범용 도구를 만들다가도 명시적으로 계속 상기하지 않으면 테스트용 코드처럼 좁아짐

즉, "계속 문제를 잡는 태도" 자체는 장점이지만, 종료 조건과 범위 관리가 없으면 프로젝트를 망가뜨리는 방향으로도 작동합니다.

## 우리가 막아야 하는 3가지 실패 패턴

## 1. 목표 망각

증상:

- 눈앞의 에러 해결이 목적이 되어버림
- 처음 요청한 사용자 가치가 사라짐
- 원래 문제보다 주변 문제를 더 오래 만짐

대표 사례:

- 로그인 UX 개선이 목적이었는데, 끝나지 않는 타입 정리로 세션을 다 씀
- 범용 툴 제작이 목표였는데, 테스트 케이스 하나를 맞추는 데 구조를 계속 바꿈

## 2. 과도하게 뾰족한 해결

증상:

- 특정 테스트, 특정 입력, 특정 화면에서만 동작
- 예외 분기와 임시 우회가 빠르게 늘어남
- 해결은 했는데 재사용성은 줄어듦

대표 사례:

- 테스트만 통과하는 목 구현
- 한 컴포넌트만 맞는 하드코딩된 유틸
- 지금 에러는 사라졌지만 다른 환경에선 깨지는 설정 수정

## 3. 범용 목표 상실

증상:

- "범용 스캐폴드"를 만들다가 어느새 "현재 테스트에서만 돌아가는 코드"가 됨
- 문서에는 범용이라고 적혀 있는데 실제 구현은 특정 예시만 지원함
- 사용자가 다시 범용성을 언급하기 전까지 AI가 이를 잊음

대표 사례:

- 예제용 env만 맞는 시크릿 로더
- 특정 프레임워크 구조만 가정한 템플릿
- 재사용 툴이라고 했지만 프로젝트 하나의 맥락에 과적합된 명령어

## 원인 분석

이 문제는 대부분 아래 4가지 때문에 생깁니다.

### 1. 목표보다 현재 상태 변화에 반응함

AI는 현재 보이는 실패 신호를 강하게 따라갑니다.
그래서 "본래 문제 해결"보다 "방금 본 오류 제거"에 빨려 들어가기 쉽습니다.

### 2. 종료 조건이 약함

"어디까지 하면 끝인가"가 없으면 AI는 계속 더 고치려 합니다.

### 3. 범용성 요구가 작업 중 계속 재주입되지 않음

범용 스캐폴드나 공용 툴 작업은 매 단계마다 "이게 특정 사례에 과적합됐는가"를 다시 확인해야 합니다.
한 번만 말해두면 중간 구현에서 쉽게 잊힙니다.

### 4. 검증이 테스트 통과에 과도하게 묶여 있음

테스트는 중요하지만, 테스트 통과가 곧 범용성이나 설계 적합성을 보장하지는 않습니다.

## 예방 원칙

이 문제를 막기 위해 아래 5가지를 기본 원칙으로 채택합니다.

## 1. 모든 작업에는 원래 목표와 종료 조건을 함께 쓴다

모든 작업은 아래 3줄로 시작합니다.

```md
Original Goal:
Non-Goal:
Done When:
```

예시:

```md
Original Goal: 여러 프로젝트에서 재사용 가능한 시크릿 로더 스캐폴드 만들기
Non-Goal: 특정 테스트 케이스 하나를 맞추기 위한 임시 우회
Done When:
- 다른 프로젝트에서도 복사해 쓸 수 있다
- 프로젝트별 env 경로만 바꾸면 동작한다
- 특정 테스트 전용 하드코딩이 없다
```

## 2. 해결보다 범위 유지가 더 중요하다

AI는 "지금 보이는 문제를 더 고치고 싶은 욕구"보다 "원래 범위를 지키는 것"을 우선해야 합니다.

즉 아래 상황이면 바로 멈춰야 합니다.

- 수정이 원래 목표보다 더 커지기 시작함
- 새로운 문제를 해결하느라 다른 영역을 건드리기 시작함
- 특정 케이스 대응 코드가 늘기 시작함

## 3. 범용 작업에는 항상 일반화 체크를 넣는다

범용 툴, 스캐폴드, 공용 모듈 작업에는 아래 질문을 검증에 반드시 포함합니다.

- 이 코드는 특정 프로젝트 구조를 하드코딩했는가
- 특정 테스트 데이터가 없으면 깨지는가
- 다른 프로젝트에 복사했을 때 이름 몇 개만 바꾸면 되나
- 특정 케이스 분기가 너무 많아졌는가

## 4. 테스트 통과와 목표 달성을 분리한다

검증은 최소 2단계여야 합니다.

- Local Verification: 현재 예시/테스트에서 동작하는가
- Goal Verification: 원래 문제를 범용적으로 해결했는가

이 둘 중 하나라도 실패하면 완료로 보지 않습니다.

## 5. 멈춤 규칙을 강제한다

AI가 아래 상태에 들어가면 자동으로 멈추고 사용자에게 재정렬을 요청하거나 로그를 남겨야 합니다.

- 같은 종류의 수정이 3회 이상 반복됨
- 목표와 직접 관련 없는 파일 수정이 늘어남
- 테스트는 통과하지만 범용성 설명이 약함
- 문제 해결보다 예외 처리 추가가 더 많아짐

## 운영 기획안

## A. 작업 시작 템플릿

모든 실작업 전에 아래 템플릿을 채웁니다.

```md
Task Type:
Original Goal:
User Value:
Non-Goal:
Scope:
Generic Requirement:
Done When:
Stop If:
```

설명:

- `Original Goal`: 본래 해결하려는 문제
- `User Value`: 이게 왜 필요한지
- `Non-Goal`: 이번에 해결하지 않을 것
- `Generic Requirement`: 범용 작업이라면 반드시 유지해야 할 일반성
- `Stop If`: 멈춰야 하는 조건

## B. 작업 중 체크포인트

큰 작업에서는 아래 시점마다 체크합니다.

### Checkpoint 1. 첫 구현 전

- 지금 이해한 목표가 처음 목표와 같은가
- 이번 수정이 진짜 필요한 최소 범위인가

### Checkpoint 2. 첫 동작 확인 후

- 지금 돌아가는 것이 "예시에서만" 돌아가는 것은 아닌가
- 테스트가 아니라 실제 목표 기준으로 설명 가능한가

### Checkpoint 3. 마무리 직전

- 더 고치고 싶은 부분이 있어도 이번 목표와 무관하면 멈출 수 있는가
- 이 상태를 "충분히 좋다"고 선언할 수 있는가

## C. 종료 판정 방식

완료 판정은 아래 3개를 모두 만족해야 합니다.

1. 원래 목표를 다시 읽어도 지금 결과가 맞다
2. 테스트 통과 외에 범용성/재사용성 설명이 가능하다
3. 추가 수정이 "품질 향상"이 아니라 "과잉 최적화"에 가깝다면 멈춘다

## D. 멈춤 선언

AI는 아래 문장을 세션 종료 전 반드시 남길 수 있어야 합니다.

```md
We should stop here because the original goal is satisfied.
Any further changes would mainly optimize edge cases or preferences, not core value.
```

이 선언을 못 하면 아직 종료 조건이 약한 것입니다.

## 이 스캐폴드에 적용할 실무 규칙

## 1. 모든 범용 작업에는 `Generic Requirement`를 적는다

예시:

- framework-agnostic
- project-agnostic
- reusable across multiple repos
- not test-only

## 2. 테스트 전용 해법은 명시되지 않으면 금지한다

다음은 기본적으로 금지합니다.

- 특정 테스트 파일만 통과하는 하드코딩
- 예시 데이터 전용 분기
- 현재 워크스페이스 경로 고정
- 현재 프로젝트 구조 고정

## 3. 완료 보고에는 반드시 아래가 포함되어야 한다

- Original Goal 충족 여부
- Generic Requirement 충족 여부
- 왜 지금 멈춰도 되는지
- 남겨둔 비목표 항목

## 4. 작업 로그에는 실패 패턴도 기록한다

`worklogs/`에 아래 항목을 남깁니다.

- 목표를 잊고 옆길로 샜는가
- 특정 케이스 과적합 징후가 있었는가
- 범용 요구를 중간에 다시 상기해야 했는가

## 오케스트레이터로 옮길 규칙

이 계획은 나중에 아래 검사들로 자동화할 수 있습니다.

- 작업 계획에 `Original Goal`, `Non-Goal`, `Done When`, `Stop If`가 있는지 검사
- 범용 작업인데 `Generic Requirement`가 비어 있으면 실패 처리
- 완료 로그에 "왜 여기서 멈추는지"가 없으면 경고
- 테스트 전용 경로/문구/하드코딩 탐지

## 추천 결론

가장 중요한 건 AI에게 "더 고쳐라"보다 "언제 멈춰야 하는가"를 가르치는 것입니다.

이 문제를 막는 핵심 장치는 아래 4개입니다.

- Original Goal 고정
- Non-Goal 명시
- Generic Requirement 반복 확인
- Stop If 강제

즉, 앞으로의 바이브 코딩 운영은 "문제 해결 중심"만이 아니라 "종료 설계 중심"으로 바꿔야 합니다.
