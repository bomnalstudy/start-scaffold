# Project Guardrails

## 핵심 원칙

- 한 번에 하나의 문제만 해결합니다.
- 기능 추가보다 회귀 방지를 우선합니다.
- 구조 변경은 근거와 롤백 전략이 있을 때만 합니다.
- 새 의존성 추가는 비용 대비 이득이 명확할 때만 합니다.

## 변경 위험도 분류

### Low

- 단일 파일 수정
- 텍스트/문서 변경
- 명확한 버그 수정

### Medium

- 여러 모듈에 걸친 로직 수정
- API 계약 변경
- 빌드 설정 수정

### High

- 인증/결제/데이터 저장소 변경
- 대규모 리팩터링
- 테스트 부재 상태에서의 핵심 흐름 수정

High 변경은 아래를 만족해야 합니다.

- 목적이 문서화되어 있음
- 영향 범위가 기록되어 있음
- 검증 방법이 정의되어 있음
- 롤백 경로가 있음

## 프로젝트가 망가지는 전형적 징후

- AI가 비슷한 파일을 계속 새로 만듦
- 에러 하나를 잡으려다 설정 파일이 연쇄적으로 바뀜
- 사용하지 않는 유틸과 추상화가 빠르게 늘어남
- "일단 돌아가게" 만든 우회 코드가 남아 있음
- 문서와 실제 구조가 달라짐

## 작업 전 질문

- 이 변경은 되돌리기 쉬운가
- 지금 바꿔야 하는가
- 더 작은 수정으로 해결 가능한가
- 검증 가능한가

## 작업 후 질문

- 복잡도가 늘었는가 줄었는가
- 문서 업데이트가 필요한가
- 다음 세션의 AI가 이 변경을 이해할 수 있는가
- 실패했을 때 어디부터 추적할 수 있는가
