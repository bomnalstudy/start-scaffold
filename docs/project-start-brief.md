# Project Start Brief

이 문서는 새 프로젝트를 시작할 때 AI와 사람이 함께 먼저 정리해야 할 질문 목록입니다.

목적은 단순합니다.

- 시작할 때 목표를 잘 묻는다
- 오케스트레이터가 그 목표를 계속 기억하게 한다
- 작업 범위를 항상 최소치로 잡는다
- "일단 크게"가 아니라 "작게 되는 MVP"부터 만든다
- 개발자 관성보다 빠른 실행과 명확한 기준을 우선한다

## 핵심 원칙

- 시작 질문이 애매하면 프로젝트 전체가 흔들립니다.
- 첫 범위는 항상 최소 MVP 수준으로 잡습니다.
- AI는 이후 모든 작업에서 처음 정한 목적을 다시 확인해야 합니다.
- 새 이슈가 보여도 현재 MVP 범위를 먼저 닫기 전에는 확장하지 않습니다.

## 이 스캐폴드의 기본 철학

이 스캐폴드는 "정교한 개발자식 완성도"를 첫 목표로 두지 않습니다.

대신 아래를 먼저 달성하는 데 집중합니다.

- 요구 조건을 명확하게 적는다
- 가장 작은 프로젝트 단위로 문제를 해결한다
- 빠르게 만들고 바로 검증한다
- 충분하면 멈춘다

즉 기준은 "얼마나 정교한가"보다 아래에 가깝습니다.

- 지금 요구를 해결하는가
- 범위가 작고 통제 가능한가
- 다음 반복으로 넘길 수 있게 닫혔는가

## 시작 전에 반드시 답할 질문

## 1. 이 프로젝트는 정확히 무엇을 해결하나

- 누구의 어떤 문제를 해결하나
- 결과물이 무엇이면 "작동한다"고 말할 수 있나
- 가장 중요한 사용자 가치는 무엇인가

예시:

- "프로젝트별 시크릿을 로컬에서 안전하게 저장하고 다른 PC로 옮길 수 있게 한다"
- "AI가 목표를 잊지 않도록 시작 브리프와 종료 조건을 강제한다"

## 2. 첫 버전에서 반드시 되는 것은 무엇인가

- 첫 버전에서 꼭 되는 기능 1~3개는 무엇인가
- 없어도 되는 기능은 무엇인가
- 나중에 붙일 기능은 무엇인가

이 질문은 곧 `MVP Scope`가 됩니다.

추가 원칙:

- 첫 버전은 "작은 프로젝트 하나"처럼 닫혀야 합니다.
- 이후 확장은 다음 iteration으로 넘깁니다.

## 3. 무엇은 이번 작업의 비목표인가

아래를 반드시 적습니다.

- 이번에 하지 않을 것
- 좋아 보이지만 지금은 미룰 것
- 다음 단계로 넘길 것

이 항목이 없으면 AI는 자꾸 범위를 넓힙니다.

## 4. 범용성은 어디까지 필요한가

범용 작업이라면 아래를 명확히 적습니다.

- 특정 프로젝트 전용인가
- 여러 프로젝트에서 재사용해야 하는가
- 프레임워크 독립적이어야 하는가
- 테스트 전용 구현을 허용하는가

이 항목은 `Generic Requirement`가 됩니다.

## 5. 어디서 멈출 것인가

아래 질문에 답해야 합니다.

- 어느 상태면 "충분히 좋다"고 볼 것인가
- 어떤 순간부터는 과잉 최적화인가
- 어떤 조건이면 범위를 다시 협의해야 하나

이 항목은 `Stop If`와 `Done When`으로 연결됩니다.

## 시작 브리프 템플릿

## 오케스트레이터가 기억해야 할 것

오케스트레이터는 아래 5가지를 세션 내내 유지해야 합니다.

- Original Goal
- MVP Scope
- Non-Goal
- Generic Requirement
- Stop If

이 5가지 중 하나라도 빠지면 방향이 흐트러질 가능성이 큽니다.

## 작업 중 반복 체크 질문

오케스트레이터는 각 체크포인트에서 아래를 다시 확인합니다.

### 방향 체크

- 지금 하고 있는 수정이 Original Goal에 직접 연결되는가
- 지금 수정이 MVP Scope 안에 있는가
- 지금 보이는 문제는 진짜 핵심인가, 옆문제인가

### 범위 체크

- 이 수정은 최소 범위인가
- 더 작은 해결로 끝낼 수 없는가
- 지금 새 파일/새 추상화가 정말 필요한가

### 범용성 체크

- 이 구현이 예시나 테스트 하나에 과적합됐는가
- 다른 프로젝트에서도 설명 가능한 구조인가

### 종료 체크

- 이미 충분히 목표를 달성했는가
- 지금부터는 개선이 아니라 과잉 최적화인가

## MVP 우선 규칙

이 스캐폴드에서는 항상 아래 순서를 기본으로 둡니다.

1. 문제를 해결하는 가장 작은 버전 만들기
2. 그 버전을 안정적으로 검증하기
3. 그 다음에만 확장하기

즉 아래는 금지에 가깝습니다.

- 처음부터 범용성, 편의성, 확장성, 자동화를 한 번에 다 잡으려는 시도
- 아직 MVP가 안 닫혔는데 리팩터링부터 하는 것
- 사용자 가치보다 구조적 아름다움을 우선하는 것

## 빠른 실행 규칙

이 스캐폴드는 아래 질문에 "예"라고 답할 수 있는 쪽을 선택합니다.

- 이 요구를 해결하는 가장 작은 결과물인가
- 오늘 안에 끝낼 수 있을 정도로 작게 쪼개졌는가
- 구현보다 요구 기준이 먼저 적혀 있는가
- 다음 단계로 넘겨도 되는 것을 욕심내어 지금 하고 있지 않은가

```md

MVP Scope:
Non-Goal:
Generic Requirement:
Stop If:
```

작업 종료 시에는 아래처럼 요약해야 합니다.

```md
What remains for the next iteration:
```
