# AI Usage Guide

## 이 스캐폴드가 막으려는 문제

바이브 코딩은 속도를 올려주지만 아래 문제가 자주 생깁니다.

- 문제 정의 없이 코드를 먼저 늘려 구조가 무너짐
- 짧은 성공을 위해 임시 패치가 누적됨
- 이전 대화 맥락이 사라져 같은 실수를 반복함
- 검증보다 생성에 치우쳐 "되는 것처럼 보이는 코드"가 쌓임
- 본래 목표를 잊고 눈앞의 문제만 끝없이 고침
- 특정 테스트나 예시만 맞는 뾰족한 해법으로 빠짐
- 범용 툴 작업인데도 중간에 범용성 요구를 잊어버림

## 권장 작업 패턴

한 세션에서 아래 흐름을 유지합니다.

0. 먼저 `docs/token-ops-standard.md`를 확인
1. `Original Goal`, `MVP Scope`, `Non-Goal`, `Done When`을 먼저 적기
2. 관련 파일만 확인
3. 구현 범위를 최소 MVP 수준으로 제한
4. 수정 후 즉시 검증
5. "왜 지금 멈춰도 되는지"까지 기록

기본 태도:

- 개발자처럼 크게 설계하려 하기보다 작은 프로젝트 하나를 빠르게 닫는 방식으로 움직입니다.
- 요구 조건이 애매하면 구현보다 기준 정리를 먼저 합니다.
- MVP가 닫히기 전에는 확장성, 편의성, 추상화를 먼저 욕심내지 않습니다.

범용 작업이라면 아래도 추가합니다.

- `Generic Requirement`
- `Stop If`

## 좋은 프롬프트 패턴

좋음:

- "`/api/auth` 로그인 실패 원인을 찾아서 최소 수정으로 고쳐줘. 수정 전 관련 파일만 먼저 확인하고, 바꾼 뒤 검증 방법까지 정리해줘."
- "`docs/project-guardrails.md` 기준을 지키면서 결제 모듈 구조를 점검해줘. 대규모 리팩터링은 하지 말고 위험한 부분만 짚어줘."

나쁨:

- "전체 프로젝트 다 뜯어고쳐줘"
- "모던하게 리팩터링해줘"
- "에러 고치고 성능도 올리고 테스트도 만들고 디자인도 바꿔줘"

## AI에게 항상 함께 주면 좋은 정보

- 이번 작업 목표
- 이번 작업의 MVP 범위
- 이번 작업의 비목표
- 수정 가능한 범위
- 수정하면 안 되는 범위
- 완료 판단 기준
- 테스트 또는 확인 방법
- 범용 작업인지 여부
- 어디서 멈춰야 하는지

## 세션 운영 팁

- 기능 구현 세션과 코드 리뷰 세션을 분리합니다.
- 긴 대화를 이어가기보다, 작업 단위를 닫고 새 세션으로 넘어갑니다.
- 중요한 결정은 문서에 남기고 채팅 기록에 의존하지 않습니다.
- 에이전트별 진입 문서는 Codex=`CODEX.md`, Claude=`CLAUDE.md`를 사용합니다.
- 새 프로젝트나 큰 작업은 `docs/project-start-brief.md` 질문으로 먼저 시작합니다.
- 범용 툴 작업은 `docs/vibe-coding-failure-prevention.md`의 종료 규칙을 함께 봅니다.
