# Orchestration Plan

## Project / Task

- Add `add-mode` as a repo-local customization mode

## User Problem

- The scaffold can narrow work by topic, but it does not yet have a dedicated mode for turning a user's persistent preferences or workflow rules into durable scaffold changes.
- The user wants a mode that decides what should be updated across `AGENTS.md`, custom skills, mode docs, and related markdown files so future sessions follow the request without repeated prompting.

## Original Goal

- Create `add-mode` as a repository customization mode that routes durable user requirements into the right scaffold layers.

## User Value

- Future sessions can reuse project-specific rules and working styles because the scaffold has a clear mode for encoding them in the right place.

## Priority

- Primary KPI (must): Time saved
- Secondary KPI (optional): Quality that reduces rework

## MVP Scope

- Add `docs/modes/add/` guidance for classifying and routing durable user requirements.
- Add a repo-local `skills/add-mode/` entry point with UI metadata.
- Update shared mode/skill docs and README references so `add-mode` is discoverable.
- Record the new mode and its assumptions in task/worklog files.

## Non-Goal

- Build an automatic rule-merging engine.
- Add host-level slash indexing or product-side skill automation.
- Rewrite existing modes to depend on `add-mode`.

## Generic Requirement

- Keep `add-mode` procedural and lightweight.
- Favor routing rules over duplicated documentation.

## Stop If

- The work expands into automation or generalized config-engine design.
- The new mode starts duplicating large portions of `AGENTS.md` or shared docs instead of routing to them.

## Pattern

- Scaffold Customization

## Roles

### Planner

- Input: current mode system and skill structure
- Output: minimal add-mode scope and routing rules

### Builder

- Input: `docs/`, `skills/`, shared mode references
- Output: new add-mode docs and skill files

### Reviewer

- Input: changed mode references and new add-mode wording
- Output: discoverability and drift review

### Verifier

- Input: task plan and worklog
- Output: focused session guard checks

### Recorder

- Input: final changes and assumptions
- Output: session worklog

## Scope

- Included: `docs/`, `skills/`, `README.md`, `worklogs/`
- Excluded: `scripts/`, host settings, external skill directories

## Risks

- `add-mode` may overlap with `failure-pattern-mode` if it focuses on mistakes instead of durable customization.
- The mode may become too abstract if it does not clearly explain where rules should live.

## Done When

- `add-mode` exists as a repo-local skill with matching docs.
- Shared references list `add-mode` as a standard mode.
- The new docs explain how to route persistent user requirements to the correct scaffold layer.

## Verification

- Review `add-mode` wording for trigger clarity and layer-routing usefulness.
- Run focused session guard checks on the plan/worklog.

## Why Stop Now

- Once `add-mode` exists and the routing rules are documented, the scaffold has the minimum reusable structure needed to encode future persistent user requests without overbuilding automation.
